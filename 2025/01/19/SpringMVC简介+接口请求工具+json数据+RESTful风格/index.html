二.接口请求工具（可根据视频教程使用PostMan，但是在实际开发中会使用更现代化的工具apifox）

Postman 使用教程

  1. 下载与安装：访问 Postman 官方网站，下载适合自己操作系统的版本，然后按照安装向导进行安装。安装完成后，打开 Postman 应用程序，可以注册或登录账号以便同步数据等。

  2. 发送第一个请求

    • 新建请求：点击 “New” 按钮，选择 “Request” 创建一个新的请求。
    • 设置请求方法和 URL：在请求编辑器中，选择要发送的 HTTP 请求方法（如 GET、POST 等），并输入请求的 URL 地址。
    • 添加请求参数：如果请求需要携带参数，可在 “Params” 选项卡中添加查询参数，以键值对的形式输入参数名和参数值。对于 POST 等请求，如果需要发送表单数据、JSON 数据等，可在 “Body” 选项卡中选择相应的类型并输入数据。
    • 发送请求：点击 “Send” 按钮，Postman 将发送请求，并在右侧显示响应结果，包括状态码、响应头和响应体等。

  3. 设置环境变量：点击 Postman 右上角的小眼睛图标，可手动设置环境变量。也可以使用脚本设置环境变量，例如通过 JSON 提取器或正则表达式提取器从响应中提取值，并设置为全局变量供后续请求使用。

  4. 接口断言：在 “Tests” 选项卡中编写断言脚本，用于验证响应的正确性。例如，检查状态码是否为 200、响应体是否包含特定字符串、JSON 值是否符合预期等。

  5. 批量运行测试用例：使用 CSV 或 JSON 格式编写接口测试用例，然后在 Postman 中通过设置集合和迭代器等方式批量运行测试用例。

Apifox 使用教程

  1. 下载与安装：访问 Apifox 官方网站，选择适合自己操作系统的版本进行下载和安装。安装完成后，打开 Apifox 应用程序，注册或登录账号。
  2. 新建团队和项目：点击 “新建团队”，输入团队名称并保存。进入团队后，点击 “新建项目”，输入项目名称、选择项目图标等，还可以选择是否包含示例数据。
  3. 项目环境配置：在项目设置中，选择一个测试环境，配置前置 URL 等信息。配置后，新建接口时只需填写接口地址即可。
  4. 新建接口：点击 “新建接口”，选择请求方式（如 GET、POST 等），填写接口请求路径、接口名称等信息。设置请求参数，包括参数类型、是否必填等。如果请求体有数据，可在 “Body” 选项卡中选择相应的格式（如 JSON、表单等）并输入数据。
  5. 测试接口：设置好请求参数后，点击 “发送” 按钮，Apifox 将发送请求并显示响应结果。也可以使用快捷请求来测试接口，快捷请求需要填写完整的 URL 地址。
  6. 保存测试用例：将测试过程中的成功用例、失败用例等保存起来，方便后续查看和管理。
  7. 其他功能：Apifox 还支持环境变量、全局变量、临时变量、动态变量、随机参数、自动化字段、Socket 接口、团队管理、导入导出、Mock、脚本开发、自动化执行、Jenkins 持续集成等功能。

三.Get请求和POST请求

1.如何在Postman或apifox里面发送get或者post请求？如何在请求工具设置请求参数？后端如何设置接收刚刚的参数？

①Postman 相关操作

  • 发送 GET 请求

    • 新建请求：点击 “New” 按钮，选择 “Request” 创建一个新的请求。
    • 设置请求方法和 URL：在请求编辑器中，从下拉菜单中选择 “GET” 方法，并输入请求的 URL 地址。
    • 发送请求：点击 “Send” 按钮，Postman 将发送 GET 请求，并在右侧显示响应结果。

  • 发送 POST 请求

    • 新建请求：点击 “New” 按钮，选择 “Request” 创建一个新的请求。
    • 设置请求方法和 URL：在请求编辑器中，从下拉菜单中选择 “POST” 方法，并输入请求的 URL 地址。
    • 设置请求体：在 “Body” 选项卡中，选择请求体的格式，如 “form-data”（用于表单数据）、“x-www-form-urlencoded”（用于 URL 编码数据）、“raw”（用于 JSON、文本等），然后输入请求体的数据。
    • 发送请求：点击 “Send” 按钮，Postman 将发送 POST 请求，并在右侧显示响应结果。

  • 设置请求参数

    • GET 请求参数：在 “Params” 选项卡中，以键值对的形式添加查询参数。例如，要添加参数 “name” 和 “value”，则在 “Key” 列中输入 “name”，在 “Value” 列中输入 “value”。
    • POST 请求参数：如果是 “form-data” 或 “x-www-form-urlencoded” 格式，在 “Body” 选项卡中以键值对的形式添加参数；如果是 “raw” 格式且发送 JSON 数据，直接在文本框中输入 JSON 格式的数据。

  • 后端接收参数（以 SpringMVC 为例）

    • 接收普通参数：在控制器方法中，可以使用@RequestParam注解来接收 GET 或 POST 请求中的普通参数。例如：@RequestMapping(value = "/test", method = RequestMethod.POST) public String test(@RequestParam("paramName") String paramValue){//处理参数}

    • 接收 JSON 参数：在控制器方法中，使用@RequestBody注解来接收 POST 请求中的 JSON 参数。例如：@RequestMapping(value = "/test", method = RequestMethod.POST) public String test(@RequestBody User user) {//处理参数}

②Apifox 相关操作

  • 发送 GET 请求
    • 新建接口：在左侧栏中，选择 “接口”，点击右侧的 “+ 新建接口” 按钮，填写接口名称和描述，选择请求方法 “GET”，输入请求 URL。
    • 发送请求：点击右上角的 “发送” 按钮，Apifox 将向指定的 URL 发送 GET 请求，并显示响应结果。
  • 发送 POST 请求
    • 新建接口：在左侧栏中，选择 “接口”，点击右侧的 “+ 新建接口” 按钮，填写接口名称和描述，选择请求方法 “POST”，输入请求 URL。
    • 设置请求体：在 “Body” 标签页中，选择参数格式，如 “form-data”、“raw” 等，输入请求体数据。
    • 发送请求：点击右上角的 “发送” 按钮，Apifox 将向指定的 URL 发送 POST 请求，并显示响应结果。
  • 设置请求参数
    • GET 请求参数：切换到 “Params” 标签页，添加查询参数，例如 “?name=value”。
    • POST 请求参数：切换到 “Body” 标签页，选择参数格式，如 “form-data” 或 “raw”，输入请求体数据。
  • 后端接收参数（以 SpringMVC 为例）：与 Postman 后端接收参数的方式相同，使用@RequestParam注解接收普通参数，使用@RequestBody注解接收 JSON 参数。

五.json数据（了解一下json格式规则）

JSON（JavaScript Object Notation）是一种轻量级的数据交换格式，易于人阅读和编写，同时也易于机器解析和生成。

数据类型

1.对象（Object）

  • 一个对象以{开始，以}结束。

  • 它由零个或多个键值对组成，键值对之间用逗号”,“分隔。

  • 键必须是字符串，值可以是字符串、数字、布尔值、数组、对象或者null。

2.数组（Array）：

  • 一个数组以[（左方括号）开始，以]（右方括号）结束。

  • 数组元素之间用逗号（, ）分隔，元素可以是任意 JSON 数据类型。

3.字符串（String）：

  • 字符串是由双引号（"）包围的零个或多个 Unicode 字符序列。

  • 可以使用反斜杠（\）进行转义。

4.数字（Number）：

  • 可以是整数或浮点数，支持科学计数法。

5.布尔值（Boolean）：

  • 只有两个值：true和false。

6.空值（Null）：

  • 表示一个空值，用null关键字表示。

格式要求

  1. 键值对

    • 键必须是唯一的，不能重复。
    • 键和值之间用冒号（:）分隔。

  2. 分隔符

    • 对象内部的键值对之间以及数组内部的元素之间都用逗号（, ）分隔。
    • 但是，最后一个键值对或数组元素之后不能有多余的逗号（在严格的 JSON 规范中），不过一些解析器允许这种情况。

  3. 大小写敏感

    • JSON 是大小写敏感的，例如true和True是不同的，true是正确的布尔值，而True会导致解析错误。

六.响应数据

在 Web 开发中，响应数据是服务器对客户端请求做出回应所返回的数据。

1.响应数据的格式

  • JSON：这是目前最常用的格式，轻量级且易于解析和生成。
  • XML：一种标记语言，常用于一些传统的企业级应用或需要严格数据结构定义的场景。
  • 纯文本：简单返回文本信息，如错误提示、简单的状态信息等。例如："User not found"。

2. 响应数据的组成部分

  • 状态码：表示请求处理的结果状态。常见的如：
    • 200 OK：请求成功，服务器已成功处理请求并返回数据。
    • 404 Not Found：请求的资源不存在。
    • 500 Internal Server Error：服务器内部发生错误，无法完成请求。
  • 响应头：包含关于响应的元信息，如Content - Type指定响应数据的类型（如application/json、text/html），Content - Length表示响应数据的长度，Cache - Control用于控制缓存策略等。
  • 响应体：实际返回给客户端的数据部分，其格式取决于Content - Type。例如，若Content - Type为application/json，响应体就是符合 JSON 格式的数据。

3. 在不同后端框架中的响应数据处理

  • SpringMVC（Java）：@ResponseBody注解可将方法返回值转为响应体，@RestController是@Controller和@ResponseBody的结合。
  • Django（Python）：视图函数用JsonResponse返回 JSON 数据。
  • Express.js（Node.js）：通过res.json()返回 JSON 格式响应。

七.RESTful风格

1.什么是RESTful风格？总结一下URL格式。

  • RESTful 风格：REST（Representational State Transfer），即表述性状态转移，是一种软件架构风格。它不是一种协议或技术，而是一种设计 Web 服务的理念。RESTful 风格的 Web 服务通过使用 HTTP 协议的一些特性，以一种简洁、可扩展的方式来暴露资源。其核心原则是将网络上的一切都视为资源，通过统一的接口对这些资源进行操作。
  • URL 格式：
    • 资源定位：URL 应清晰地定位到具体资源。例如，/users表示用户资源集合，/users/1表示 ID 为 1 的用户这一具体资源。
    • 名词使用：URL 中尽量使用名词，避免使用动词。比如，使用/products而不是/getProducts。
    • 版本控制：可以在 URL 中包含版本信息，如/v1/users，这样便于对 API 进行版本管理和升级。
    • 层级关系：如果资源有层级关系，通过路径分隔符体现。例如，/stores/1/products表示 ID 为 1 的商店下的产品资源。

2.RESTful有几种行为操作（get、post、delete、put、delete），分别对应了对资源哪种操作？

  • GET：用于获取资源。可以获取单个资源，如GET /users/1获取 ID 为 1 的用户信息；也可以获取资源集合，如GET /users获取所有用户信息。
  • POST：用于创建新资源。例如，POST /users，请求体中包含新用户的信息，服务器接收到该请求后创建一个新用户。
  • PUT：用于更新资源。通常用于更新整个资源，如PUT /users/1，请求体包含完整的用户信息，服务器用请求体中的数据完全替换 ID 为 1 的用户的现有数据。
  • PATCH：也是用于更新资源，但与 PUT 不同，它通常用于部分更新。例如，PATCH /users/1，请求体只包含需要更新的字段，服务器只更新这些字段。
  • DELETE：用于删除资源。如DELETE /users/1删除 ID 为 1 的用户。

3.思考一下，如何将所有的http请求抽象成对资源的增删改查操作？如何将对资源的操作抽象成restful形式

  • 确定资源：首先明确应用中有哪些资源，比如一个电商应用可能有用户、商品、订单等资源。
  • 设计 URL：按照 RESTful 的 URL 设计原则为每个资源设计 URL。例如，用户资源可以是/users，商品资源是/products。
  • 映射操作到 HTTP 方法：
    • 对于资源的获取操作，使用 GET 方法。如获取商品列表是GET /products，获取单个商品是GET /products/{productId}。
    • 创建资源使用 POST 方法，如POST /orders创建新订单。
    • 更新资源根据情况使用 PUT（全量更新）或 PATCH（部分更新），如PUT /users/{userId}更新用户信息，PATCH /users/{userId}部分更新用户信息。
    • 删除资源使用 DELETE 方法，如DELETE /comments/{commentId}删除评论。
  • 状态码使用：在响应中合理使用 HTTP 状态码。例如，创建成功返回 201 Created，获取成功返回 200 OK，删除成功返回 204 No Content，资源不存在返回 404 Not Found 等，以准确反映操作的结果。
